EOS.IO Software【软件】 Roadmap【蓝图】

作者： Lian

日期： 2018年05月21日 20:05

This document outlines the development【开发】 plan【计划】 from a high level【宏观】 and will be updated【更新】 as progress【进度】 is made toward version 1.0. It should be noted that this roadmap applies only to the blockchain software and not to the other tools and utilities such as wallets【钱包】 and block explorers【观测器】 which will have their own teams and dedicated【专门】 roadmaps once Phase 1 is complete.

Everything contained in this document is in draft【草稿】 form and subject to【旨在】 change at any time and provided for information purposes only. block.one does not guarantee【保证】 the accuracy of the information contained in this roadmap and the information is provided “as is” with no representations or warranties, express or implied.

Phase 1 - Minimal【最小】 Viable Testing Environment - Summer 2017

The goal of this phase is to establish【建立】 the APIs【接口】 that developers will require to start building【构建】 and testing【测试】 applications on EOS.IO. In order for developers to start testing their applications they will require the following to be implemented【实现】:

Standalone【独立】 Node【节点】 (Dan & Nathan)

A standalone node operates【处理】 a test【测试】 blockchain and produces blocks while exposing【暴露】 an API. This node does not need to concern【关心】 itself with any P2P networking【网络】 code.

Native【本地】 Contracts【合约】 (Nathan)

The EOS.IO software has a number of native contracts. These are contracts that manage【管理】 the core【核心】 operations【操作】 of the blockchain and exist outside the Web Assembly interface【接口】. These contracts include:

@eos - manages EOS token【令牌】 transfers【转换】
@stake【股权】 - manages locked【锁定】 EOS, voting【投票】, and Producer Election【选举】
@system - manages permissions【权限】, messages【消息】, and contact code updates【更新】

Virtual Machine【虚拟机】 API (Dan)

Contracts are compiled【编译】 to WebAssembly (WASM) and WASM must interface with【通信】 the blockchain via a defined【定义】 API. This API is what developers depend upon【依赖】 to build applications and be relatively stable【稳定】 before developers can really start to build on EOS.

RPC Interface【接口】 (Arhag, Nathan)

A simple【简单】 JSON RPC over HTTP interface【接口】 will be provided that enables developers to broadcast【广播】 transactions and query【查询】 application state. This is critical for both publishing【发布】 and interacting with【交互】 test applications.

Command line Tools【命令行】 (Arhag)

Command line tools facilitate integrating【整合】 the RPC interface with developer build environments.

Basic【基本】 Developer Documentation【文档】 (Josh)

Documents that teach【指导】 developers how to get started with building on EOS.IO blockchains【区块链】. This includes documentations of the WASM API, RPC Interface, and Command Line Tools.

Phase 2 - Minimal Viable Test Network【测试网络】 - Fall 2017

Everything in Phase 1 assumes a trusted【可信任】 environment that only runs the developer’s own【自己】 code. Before a test network can be deployed several additional features need to be implemented and tested.

P2P Network【网络】 Code (Phil)

This is a plugin【插件】 that is responsible for synchronizing【同步】 the blockchain state between two standalone【独立】 nodes.

WASM Sanitation & CPU Sandboxing【沙盒】 (Brian)

The WASM code needs to be sanitized【清洁】 to check for non-deterministic behavior such as floating point【浮点】 operations and infinite【无限】 loops.

Resource【资源】 Usage Tracking【跟踪】 & Rate【速度】 Limiting (Arhag)

To prevent【防止】 abuse【滥用】 the resource monitoring【监控】 and usage tracking rate limits【限制】 users accoding to staked【股权】 EOS.

Genesis【起源】 Import Testing (DappHub)

Tools need to be developed to export【导出】 data from the EOS Token Distribution【分布式】 state and create a genesis【初始】 configuration【配置】 file. This will enable anyone participating【参与】 in the Token Distribution to acquire【获取】 some initial test EOS (TEOS).

Interblockchain【区块链内部】 Communication【通信】 (Nathan)

This feature involves verifying【校验】 the Merkle hashing【哈希】 of transactions【交易】 is proper.

Phase 3 - Testing【测试】 & Security【安全】 Audits【审计】 - Winter 2017, Spring 2018

During this phase the platform will undergo【承受】 heavy testing【测试】 with a focus on finding security【安全】 issues and bug. At the end of Phase 3 version 1.0 will be tagged.

Develop Example【示例】 Applications

Example applications are critical to proving【证明】 the platform provides the features【特性】 required by real developers.

Bounties【奖金】 for Succesfully Attacking【攻击】 Network

Attacking the network with spam【垃圾邮件】, virtual machine exploits【利用】, and bug crashes, and non-deterministic behavior【行为】 will be a heavily involved process【过程】 but necessary to ensure that version 1.0 is stable.

Language【语言】 Support

Adding support for additional【其他】 langauges to be compiled to WASM: C++, Rust, etc.

Documentation【文档】 & Tutorials【教程】

Phase 4 - Parallel【并行】 Optimization【优化】 Summer / Fall 2018

After getting a stable 1.0 product released, we will move toward optimizing【优化】 the code for parallel execution【执行】.
